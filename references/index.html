Platform migration to Azure Kubernetes Service

Migrating a complex Tectonic environment to AKS

AKS

...

Platform migration to Google Kubernetes Engine

Migrating a complex VM environment to GKE

GKE

...

In this case, we were helping the customer to migrate their virtual machine-based environment to the public cloud and build a reliable, convenient development environment with a modern tool set. They had already been using Google Cloud Services, so it was natural to use this platform in the future as well. We also planned a complete solution for logging and monitoring (Loki, Prometheus, Grafana), to help the client reveal the possible bottlenecks and problems.

Operations support for a Kubernetes platform

Supporting a custom Kubernetes environment

Kubernetes

...

The tasks in this case consisted of integrating multiple operations support solutions into an existing, frequently changing on-premises development Kubernetes platform. The overall goal was making the platform stable, and also making the processes and resources transparent for the client, therefore, these operations support services were mainly related to logging, monitoring and alerting to help the client get a better understanding of what is happening to their applications inside the cluster, and to identify the failing components and bottlenecks.

We performed the integration of the most popular available monitoring and logging solutions (Prometheus, Grafana, ELK stack) along with finding and fixing faulty configurations of some of the existing components. One of the major tasks included the implementation of an API gateway from scratch to make communication possible between different alerting systems. We also helped to fix problems that occurred when configuring persistent storage behind the clusters (MinIO). We have also been providing support for these systems since the beginning of the project.

Platform migration from Amazon AWS

Optimizing the platform to manage the costs

Kubernetes

...

Our customer originally built an OLTP auction system for mobile ad serving, running in Amazon AWS, based on ECS. The solution provided by the application inherently meant a large amount of network traffic (lots of small transactions), which after some substantial business growth sparked the idea to optimize the cloud provider costs and find a hybrid solution, moving the most network-intensive parts to on-premises, as there was a huge gap in the network costs of the different providers. As the application was originally architected to use different kinds of AWS services, this also meant making changes to be able to run in a Kubernetes environment.

Because of the contract terms, there was a huge financial incentive to finish the move in time, and since the customer had limited amount of cloud engineer resources available, we were involved in the design and implementation of the hybrid solution, involving a self-maintained on-premises Kubernetes cluster. The nature of the application meant that very strict response times had to be ensured, so a part of the project was to help the customer selecting a CNI solution that can fulfil the requirements.
